# 4.1. 제약 기반 사고
- 극복 해야할 과제, 그 결과 급진적인 미래 지향적 발명품 탄생
- 이러한 도전은 **제약**으로 볼 수 있음
- 창의성을 발휘하는데 있어, **제약**은 가장 중요한 요소
	- 하나의 캔버스에 고집스럽게 그렸던 세잔
- **제약 기법**을 사용하여 진정으로 독특한 예술 작품을 만들어냄
- 반면, 제약은 **강요된 것 일 수 있음**
	- 이 경우에는, 주어진 것을 가지고 작업하거나
	- 그 제약에서 벗어날 방법을 찾아야 함
- 제약의 영향을 받은 프로세스의 최종 결과는 **매우 진보적**일 수 있음
	- 제약 x -> 창의성 x
- [[요한 볼프강 폰 괴테]] 
- 레거시 코드나 조잡한 기존 데이터베이스 시스템으로 작업해야만 할 때가 온다면 이점을 기억할 것

### 4.2.2. 소프트웨어 제약 조건

	- 내재적 제약은 원하든 원하지 않든, 예술적 결과물을 만들어내는데 도움이 됨
- e.g. 페인트의 종류에 따라, 고유한 모양, 그리는 방법, 수분의 양, 내광성, 일관성 등에 따른 지속성에 따라 다름
	- 아크릴 물감은 빨리 마르고 불투명함
		- 어두운 색에서 밝은 색을 칠할 수 있지만,
	- 수채화는 그 반대
- 소프트웨어도 같은 원리
	- 소프트웨어 개발 생태계에서의 본질적인 범위 내 작업 필요
	- e.g. 키보드와 같은 주변기기 활용, 컴파일을 위한 명령 수행 등
- 컴퓨터에 작동 방법을 알려주려면 일련의 지침이 필요
	- 소프트웨어 개발의 본질적 요소
	- 프로그래밍 언어는 개발자가 자율적으로 선택할 수도, 아닐 수도 있으나(자율적 제약)
	- 더 큰 차원에서 소프트웨어 개발 본질에 부합해야 함
- EXERCISE

		- e.g. 임베디드 os는 자체적으로 스레드 지원하지 x

# 4.3. 부과적 제약 조건
- 내재적 제약: 작업하기로 선택한 자에 내재되어 있는 제약
- 부과적 제약
	- 이해당사자로부터 발생하는 제약
	- 내가 자료를 선택하는 것이 아니라 다른 사람이 선택한다는 점이 다름
- 일반적인 프로젝트의 제약이 이 범주의 속함
	- 고객은 빠르게 해결하기를 원함
	- 유지보수 팀의 언어에 맞춰서 처리
	- 예산, 시간, 효율성, 관련성 등
- 내재적 제약 및 부과적 제약의 차이는 미미해 보이나,
	- 그 심리적 결과는 팀에 큰 영향을 미침
- 내재적 제약: 거의 불평하지 않음
- 창의성을 발휘하면 관련된 모든 사람의 공통의 의지가 필요
- 부과적 제약 조건에 대해 **부정적 관점**을 긍정적으로 바꾸도록 노력하기
	- 리소스가 제한되어 있고, 아이디어가 쉽게 거부되고, 그래서 창의성이 억압된다 x
	- 제한된 리소스로 작업할 수 있고, 아이디어는 좀 더 숙고할 필요가 있으며, 창의성은 당연한 것을 넘어선다는 생각으로 접근하기
- EXERCISE
	- 현재 어떤 제약 조건에 따라 작업해야 하나요?
	- 이러한 제약이 **내재적**이지 않고 **부과적**인 조건인 이유는 무엇이라고 생각하나요?
	- 어떤 제약 조건이 여러분을 고민하게 만드는가?
	- 여러분과 여러분의 팀은 이러한 제약에 어떻게 대처하는가?


# 4.4. 스스로 부과한 제약 조건
- 제약 기반 사고 -> 창의성을 촉발하는 흥미로운 방법
	-  but, 창의성 및 디자인 연구자들은
	- 내재적 제약과 부과적 제약을 여전히 **부수적인 제약 조건**으로 간주
- 스스로 부과한 제약은 **본질적인 제약 조건**으로 볼 수 있음
	- 창의력을 높이고, 독창적인 작품을 발명하는 등
	- 기대되는 이익을 위해 의도적이고 자발적으로 부과
- 마이클 모세 비스크야르(Michael Mose Biskjaer)의 논문
	- 1920년대 전후의 수많은 고전, 현대의 아방가르드 운동은
		- 신성한 영감과 천재성이라는 개념을 버린체
		- 스스로 부과한 제약에 기반하여
		- 매우 혁신적이고 효율적인 여러 가지 창의적 개입 기법을 개념화, 고안, 적용
	- 장애물, tripwire, 명령어, 무작위 입력 자극 등을 의도적으로 자체 설정한 많은 예술가는
		- 이러한 전략이 창작 과정을 촉발하고 자극하는데 상당히 도움이 된다는 사실을 앎
- **빠듯하지만 합리적인 제약** <> 터무니 없이 말도 안되는 제약
	- tripwire는 전자에 속함
- 코딩과 관련하여 산만함을 차단하고 생산성을 높이기 위해 엄격하게 일상을 유지하는 것
	- 역시 전자에 속함
- 인터넷 액세스를 금지하거나 작업 공간을 깨끗한 흰색 화면으로 바꾸는 것과 같은 생산성 도구
	- 의도적으로 방해요소 차단, 작업 완료할 수 있도록 도와줌

# 4.5. 스위트 스폿 찾기
- **제약 조건**은 창의성을 위한 비옥한 토양
	- 단, 너무 적거나 많으면 문제가 됨
- 제약조건의 최적 조건 - 스위트 스폿(sweet spot)
	- 영감: y축, 제약 x축
	- 역 U자 형태를 가짐
	- 제약의 적당한 지점에서 영감이 제일 높게 나타남
	- 제약 부족과 제약 초과의 사이
- 제약 흐름
	- 너무 적으면, 지루한 작업 반복
	- 너무 많으면, 당면한 과제가 너무 어렵게 인식되어 학습 과정 방해
- 영감
	- 너무 적으면, 작업할 아이디어가 너무 없음
	- 너무 많으면, 혼란스럽고 모든 아이디어를 잡아야하는데 시간이 없음
- 비스크야르는 영감의 원천에 대해서 언급하면서
	- 2장에서 언급한 [[개인 지식 관리]] 시스템, 영감 엔진을 암시
- 의도적으로 스위트 스폿에 도달하고 x 펙터를 높이려면
	- **스스로 부과한 제약을 유도해야 함**
- 제약 압력을 완화하고 x를 줄이려면
	- 일시적으로 제약을 제거하거나 무시하는 등의 방법으로 제약을 줄이면 됨
	- 균형을 맞추는 것이 중요
- 이론적인 스위트 스폿은 현실과 완전히 일치하지는 않음
	- 초보자와 전문가 간에도 제약을 다르게 처리할 가능성이 높음
- 제약조건의 한계는 개인적이며 지속적인 변화에 내재되어 있다는 점을 명심해야 함
- 팀 기반의 스위트 스폿 값은 **개인의 균형점**에 근접하지 않거나, 개인의 한계를 초과할 수 있음

## 4.5.1. 적절한 제약 조건으로 추상화 촉진하기
- 문제 해결에서 적절한 수준의 제약으로 **추상화 능력**을 향상 시킬 수 있음
- **추상화**는 효율적이고 창의적인 문제 해결에 핵심 역할을 함
- 추상화에 대한 분류법

### 달걀 꾸러미 문제
- 달걀이 6개 or 8개 들어갈 수 있는 두개의 꾸러미
- 정확히 N개의 달걀을 구매하려면 몇 개의 꾸러미를 사야할까?
- 정확히 N개의 달걀을 사는게 불가하다면 `-1`을 반환해야 함
- 이 문제는 `minCartons(N int) int`를 구현해야 함
	- `minCartons(20) = 3`
	- `minCartons(7) = -1`
- 6x + 8y = N
	- x = (N-8y)/6
	- x + y = ?
	- (N-8y)/6 + y = Z
	- N-8y + 6y = 6Z
	- N-2y = 6Z
	- Z = (N-2y)/6
	- 개수 = (20-2y)/6
		- 개수 = (10-y)/3
	- 3=(10-y)/3
	- 1=
- 6x + 8y = 20
	- 6x = 20 - 8y
	- x = (20-8y)/6
	- x=(10-4y)/3
	- 2(10-4y) + 8y = N
	- 20 - 8y + 8y = N
	- 20 = N
- level-0
	- 제약 조건을 생각하지 않는다면 brute-force로 해결 가능
		- 이는 level-0라고 함
- level-1
	- 특정 사례를 찾고 이들을 다른 방식으로 처리한다면?
	- 제약 조건 1번과 추상화 레벨 1을 사용하기
	- 특수 사례를 사용해 문제 공간을 과제에 맞게 조정 가능
	- 달걀 문제 고려시 상한선 혹은 하한선이 있는가?
		- 있음. 정확히 N개의 달걀을 구매해야 하므로 6보다 작은 값은 불가능 함
- level-2
	- 제약 조건 2
	- 패턴을 찾아보기. 추상화 레벨 2에 도달
	```md
	12 {6,6} -> 14 {8,6} -> 16 {8,8} -> 18 {6, 6, 6}, ...
	```
	- 특이한 점
		- 달걀 6개들이 꾸러미가 한개 있다면 8개 꾸러미로 교체
		- 그러지 않을 경우 8개 꾸러미 2개를 6개 꾸러미 6개로 교체
	- 패턴에 대해 모든 숫자에 동작하는지 테스트
- level-3
	- 제약 조건3
	- 루프 없이 풀 수 있는 방법?
	- 수학적 추상화
	- 발견된 패턴이 규칙적인 경우 일반적으로 단계별로 값을 생성하는 대신
		- 모델에서 수학적으로 설명 가능
		- 물론, 유지보수와는 별개임
- level-1, level-3 일부를 사용하는 솔루션
```js
func minCartons(n int) int {
	switch {
		case n < 6 || n == 10: return -1
		case n % 8 == 0: return n/8
		default: return n/8 + 1
	}
}
```
- 루프를 포함하지 않아야 한다는 제약조건이 없었으면 위 답을 유추할 수 없음
- 또 다른 기법: 가능한 모든 시나리오를 포괄하는 단위 테스트를 먼저 작성하는 것
	- 최적 지점을 향해 나아가는 (스스로 부과한) 제약으로 간주 가능
- 테스트 사례를 고려하면 프로덕션 코드에서 유용하게 사용할 수 있는 패턴을 자동으로 발견할 수 있음

	- 그렇다고 무조건 좋은 것은 아님
	- [[에르베 기베르]]
- [[바니타스 그림]]

	- 자기 자신뿐만 아니라 가까운 사람들에게도 영향을 미칠 수 있음
