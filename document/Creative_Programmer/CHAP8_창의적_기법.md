# 8.1. 창의적인 도구 상자 채우기
- [[마르쿠스 아우렐리우스]]의 명상록
	- 카이사르화되거나 권력에 물들지 않도록 조심해야겠다
	- 단순하고, 선하고, 순수하고, 진지하고, 소박하고, 정의의 친구가 되고
	- 신을 두려워하고, 친절하고, 애정이 넘치고, 적절한 일을 위해 강해져야 한다
	- 철학이 만들고자 하는 사람이 되도록 열심히 노력해야 한다
- 인덱스카드가 없었다면 [[블라드미르 나보코프]]는 작가적 블록에서 회복하지 못할수도 있었음
	- 니콜라스 루만의 제텔카스텐 시스템과 마찬가지로
	- 나보코프의 카드는 창의력, 생산성을 모두 높여줌
- 민첩한 회고를 촉진하는 창의적인 도구가 없다면 2주에 한 번씩 열리는 회의는 다른 회의와 마찬가지로 지루하고 쓸모없는 시간 낭비로 끝남
	- 앞서 언급한 방법 중 최소 한 가지 방법을 사용한다면
	- 모든 전문가 모임이 지향해야 하는 즐겁고 효과적인 회의 유지 가능
- 창의적인 장애물을 극복하기 위해서는 프로그래밍이 아닌
	- 이러한 도구에 대한 기술 지식의 기초가 필요하다는 것
	- 개인 지식 관리 워크플로, [[확산 모드 사고|확산적 사고]], 스스로 부과한 제약은 창의적인 기법
	- 물론 맞지만, 포커스 그룹 연구에서 이 개념은 서로 얽혀 있으면서도 별개의 주제로 나타남
- 예술과 글쓰기는 둘다 놀라울 정도로 코딩과 유사함
	- 예술은 주어진 제약 조건 내에서 무에서 유를 창조하는 것이기 때문
	- 자유의 정도에 따라 이런 일은 외부 관행에 따라 일이 이루어지는 서류 들이밀기나 조립 라인 노동과는 다를 수 있음
	- 아티스트도 프로그램 작성자와 마찬가지로 보편적인 물리 법칙을 따를 필요는 없음
- '코딩은 예술과 유사하다'일 뿐, 코딩과 그 결과물은 예술이다는 아님
- 코딩은 글쓰기와 유사
	- 프로그래머는 자신이 하려는 바를 컴파일러가 이해하도록 구조화된 방식으로 프로그래밍
	- 동시에, 동료나 최종 사용자도 이해할 수 있게 만들어야 함
	- 컴파일러는 `buyNewBook()`이라는 함수명에 신경쓰지 않고, 그 뒤 `() {... }`에 신경씀
	- 하지만 가독성은 구문 분석, 성능보다 더 중요함
- 상상할 수 있는 모든 학문에는 고유한 기법이 있으며, 대부분은 학제간 융합 기술
	- 소프트웨어의 경계를 넘어서는 건 좋으나 그 범위가 지나치게 넓음
	- 따라서 이 책에서는 창의성 도구와 관한 조사를 아티스트, 작가, 프로그래머의 도구 상자로 제한함

### 문제 해결 기법 대 창의적 기법
- 문제 해결 기법은 언제 창의적인 기법으로 간주되는가
	- "왜"라는 똑같은 질문을 반복하는 것처럼 단순한 게 정말 창의적인가
- 창의적 기법이란 창의적인 행동을 장려하는 기술과 방법
	- 여러분과 동료들이 창의적이라고 생각한다면, 그게 무엇이든 간에 창의적이라는 점을 기억할 것
- 그렇기 때문에 지금 8장은 어떤 레시피 유형이 아님
	- 공감을 불러일으킬 수 있는 내용이 있으나, 억지스럽게 들릴 수 있음
- 오히려 대신 이 창의적 기법들을 참고하여 자신만의 기법을 생각해보기

# 8.2. 선택: 아티스트의 도구상자
- **창의적 기법** : 문맥없이 접할 때, 미술 기법들을 상상하게 됨
	- 안료, 비밀 재료를 격렬하게 혼합
	- 캔버스보다 캔버스 옆 바닥에 물감 뿌리기: 잭슨 폴록의 드립 기법
	- 수채화를 의도적으로 번지게 하는 wet-on-wet 기법
- 창의적 프로그래머에게는 큰 도움이 되지 않을 것
	- 어쩌면 도움이 될 수도
	- 조금만 노력하면 더 높은 수준의 관행, 기법, 습관이 프로그래머에게 유용한 도구가 될 수 있음

## 8.2.1. [[예술 기반 학습]]
- 예술 기반 학습은 실용적인 질문에 도움이 되지 않음
	- 인생에 더 큰 철학적 질문에 대처하는 데 도움을 주기 위한 것
	- 프로그래머로서 어떻게 하면 창의적인 문제 해결 활동과 통합할 것인가
- 이 기법은 [[앙리 푸앵카레]]의 잠재의식 창의성 이론과 매우 유사
	- 질문 숙고 후 예술 작품의 도움을 받아 잠재의식이 제 역할을 하도록 내버려 두면 창의성 발현
	- [[필립 올리버]] 쌍둥이가 개발하는 동안 접했던 텔레비전 쇼
	- 우화, 설화 또는 [[지그문트 프로이트]]를 둘러싼 물건 등
- 예술은 치료 효과가 있음
	- 예술에 대한 감상은 영감을 불러일으킴 -> 개인적인 장애물 제거에 도움
- 저명한 철학자 알랭 드 보통, 미술사학자 존 암스트롱 '알랭 드 보통의 영혼의 미술관'에서
	- 예술이 유용하고 관련성이 있으며 치유적일 수 있다고 주장
	- 예술을 바라보는 새로운 시각 제시
	- 위대한 작품은 일상생활의 긴장과 혼란을 관리할 수 있는 단서 제공 -> 이는 [[예술 기반 학습]]과 매우 흡사함
- 유명한 그림 밀크메이드(Melkmeisje)
	- 레디스 캐시를 어떻게 무효화 할 수 있는가에 대한 답은 줄 수 없겠으나
	- 예술과 우리 자신을 더 잘 이해하는 데 도움이 될 수 있으며
		- 나아가 캐시 문제에 대한 단서도 제공할 수 있음

## 8.2.2. 훔쳐라, 아티스트처럼
- 오스틴 클레온(Austin Kleon)의 도발적인 선언문
- 자신이 예술가로서 처음 시작했을 때 들었으면 좋았을 10가지를 요약함
	- 훔쳐라, 아티스트 처럼
	- 그냥 시작해라, 너무 깊이 생각하지 말고
	- 당신이 써라, 당신이 읽고 싶은 책
	- 두 손을 써라
	- 곁다리 작업이나 취미가 중요하다
	- 멋진 작업을 하고 사람들과 공유하라
	- 지리적 한계는 더 이상 없다
	- 호감형이 돼라
	- 질릴 만큼 꾸준히 하라
	- 크리에이티브는 빼기다
- 앞에서 소개된 7가지 창의적 문제 해결방법과 비슷한점이 있는가
	- '빼기' -> 제약 조건 작업
	- '질릴만큼 꾸준함' -> 끈기있게 흐름을 따라가며 근성을 발휘하는 것
		- 호기심, 친절, 체력, 바보 처럼 보이는 의지가 필요함
	- '작업공유' -> [[CHAP4_제약_조건]]의 중심 주제
- '훔쳐라 아티스트처럼'은 2장 도입부에 등장
	- 코틀린 프로그래밍 언어가 의도적으로 기존 거인들의 어깨 위에서 만들어졌고
	- 세네카가 철학적 경쟁자였던 에피쿠로스의 저서를 자주 들여다보고 배웠다는 사실을 알게 됨
	- 이 책도 같은 방식으로 만들어짐
- 클레온에 따르면 훔치는 일에는 '좋은 도둑질'과 '나쁜 도둑질'이 있음
	- 좋은 도둑질: 존경하고, 연구하고, 많은 사람으로부터 훔치고, 다른사람의 공로 인정, 변형, 재가공
	- 나쁜 도둑질: 비하, 탈세, 한 사람 것만 훔치기, 표절, 모방, 갈취
- 소프트웨어 개발 업계, 나쁜 도용이 일상적으로 이루어짐
	- e.g. MS Github Copilot 프로젝트 -> Your AI pair programmer
	- 기발한 아이디어 같으나, 실시간으로 코드의 전체 기능을 제안
		- 이는 Github Project에서 직접 가져온 수십억 줄에 달하는 코드의 라이선스 고려 X 학습
		- 어떤 종류의 승인도 제시되지 않음
	- 코파일럿은 폐쇄적인 영리 제품
		- 오픈 소스 개발자의 수천 시간에 달하는 코딩이 들어간 시간을 비윤리적으로 이용
		- 대부분 적절한 저작권 표시가 필요한 라이선스를 무시함으로써 가능해짐
	- 결국 소프트웨어 자유 단체(Software Freedom Conservancy)는 '깃허브를 포기하라'는 메세지를 발표
		- 그 결과 많은 오픈소스 소프트웨어가 Codeberg, Source Hut, Gitea 등으로 옮겨감
		- 그래도 해당 빅 테크 회사가 '무료' 데이터는 여전히 이용함
- 이런 일은 자주 일어남
	- `LICENSING.md` 파일을 자세히 살펴보지도 않고 프로젝트 의존성으로 기꺼이 추가하는 경우가 많음
	- 한 회사에서는 어떤 개발자가 GPL 소프트웨어를 사용하면서도 제품을 비공개 소스로 판매하는 것에 언급했다가 비웃음을 사기도 함
- 가장 좋은 부분을 '도용'하고 리믹스 하는 것은 종종 도메인을 발전시키기 위해 필요한 일
	- 이는 예술 뿐만 아니라 자동차, 소프트웨어 개발, 하드웨어 엔지니어링 산업에도 보임
	- 리믹싱은 때로는 단조롭고 잊기 쉬운 모방품을 만듦
	- 하지만 때로는 전혀 어울릴 것 같지 않은 조합을 리믹스해서 이전에는 상상도 못한 결과물이 나오기도 함

### EXERCISE
- 마지막으로 '예술가처럼 훔쳐본' 때가 언제인가
- 나쁜 도둑질이었는가, 좋은 도둑질이었는가
- 공부했는가 아니면 대충 훑어보았는가
- 아마 지금이 프로젝트 의존성을 빠르게 확인하고 올바르게 인증해야 할 때
- 많은 프로그래밍 에코시스템에는 노드의 경우 라이선스 검사기(license-checker)가 존재함
	- Go의 경우에는 go-license,
	- gradle에는 gradle-license-plugin
	- 엘릭서에는 licensir
- 모든 라이선스가 서로 호환되는 것은 아님

## 8.2.3. 휴식의 힘
- 그래픽 디자이너 슈테판 스타그마이스터(Stefan Stagmeister)
	- 7년마다 스튜디오를 떠나 1년동안 안식년을 가짐
	- 장기간의 휴식동안 이는 스펀지처럼 모든 것을 흡수
	- 모든 인상이 향후 창작 작업의 기초를 형성
	- 그가 방문한 몇몇 장소는 자연스럽게 멋진 영감을 불러일으킴
- 영감만을 위해 1년 내내 휴가를 내려면, 재정적 안정과 많은 베짱 필요
	- 휴가의 힘에 대한 TED 강연에서 3가지를 구분함
		- 직업(돈을 위해 9시에서 5시까지 일하는 것)
		- 경력(사다리를 오르는 것)
		- 소명(본질적으로 성취감을 느끼는 것)
	- 우리가 진정으로 원하는 것이 무엇인지 잊어버릴 때가 많다고 하면서
		- 정기적인 휴식을 통해 업무 전략을 재고하고 영감을 얻으면
		- 우리가 하는 일을 직업이 아닌 소명으로 인식할 가능성이 커진다고 주장함
- 7년마다 1년을 쉬면 12.5%의 시간을 자신이 진정으로 원하는 것을 추구하는데 할애할 수 있음
	- 정말 그렇게 많은 시간인가?
	- 구글의 이전 20%나 3M의 15% 규칙과 비교하면 실제로는 더 적음
	- 물론 이러한 '쉬는 시간'은 실제로 쉬는 것이 아니며 항상 비즈니스 수익을 염두해 두는 시간
	- 2011년부터 구글의 규모가 기하급수적으로 커지면서 비즈니스의 운영 측면에 더 집중하기 위해
		- '[[화살을 덜 만들면 더 많은 나무가 남는다]]'라는 전략을 채택하고
		- 이러한 '자유시간'을 줄이기 시작
- 안식년을 갖는다는 것은 일을 하지 않는 것이 아니라, 자신이 원하는 일을 한다는 것을 의미
	- 안식년은 거의 항상 본업에 대한 영감을 불러일으키는 계기가 됨
	- 심리학자인 대니언 길버트(Deniel Gilbert)를 비롯한 많은 작가가 안식년 중에 책을 집필
	- 대니얼 길버트는 운 좋게도 종신 교수로 재직중이어서 안식년 기간 동안 더 많은 일이 가능했음
- 슈테판은 그 기간 동안에도 여전히 예술 작품을 디자인하고 판매함
	- 심리학자 중에는 4개월 동안 진료실을 닫은 채 오직 생각하고, 글을 쓰고, 수련회를 조직하고, 영감을 얻기 위해 안식년을 갖는 사람들도 있음
	- 영감이 떠오를 수 있도록 속도를 늦추는 것은 장기 휴가의 주요 이점 중 하나
	- 심지어 스타그 마이스터가 목격한 것처럼 이후 몇 년 동안 창작의 원동력이 될 수도 있음
	- "첫번째 안식년 이후 7년 동안 우리가 디자인한 모든 것은 그 안식년에 시작되었다"
- 안식년을 갖는다고 해서 아무 계획도 세우지 않는 것은 아님
	- 스타그 마이스터는 첫 안식년은 재앙과도 같았다고 설명함
	- 아이디어를 창출하기는 커녕, 그는 그저 이런 저런 생각들을 깊이 있게 생각하지 않고 흘려보내며 반응
	- 흥미로운 일의 목록을 작성하고 이것을 실행 가능한 계획으로 전환하는 것이 더 나은 접근 방법
- 하지만 스타그마이스터가 안식년을 통해 얻는 가장 중요한 것 -> 자신의 일을 다시 사랑하게 되었다는 것
	- 그가 원망하던 직업이 진정한 소명이 됨
	- 감정은 우리가 인정하고 싶은 것보다 창의력에 더 많은 영향을 미침
	- 일 때문에 아침에 일어나야 한다는 생각을 견딜 수 없다면
		- 창의적인 돌파구가 절실하게 필요함에도, 그것을 찾을 가능성은 거의 없음
	- 심지어 번아웃으로 인해 곧 터질 수도 있다는 분명한 신호일지 모름
- 지난 수십 년 간 심리학 연구 결과는 **감정과 기분이 창의력과 분석적 문제 해결을 포함한 인지 능력**에 깊은 영향을 미친다고 주장함
	- 최근 소프트웨어 공학 연구에서는 이런 사실이 검증되지 않았으나,
	- 대니얼 그라치오틴과 그의 연구팀 -> 행복한 소프트웨어 개발자가 문제를 더 잘 그리고 창의적 해결 발견
	- 외부 프로세스와 자신의 웰빙 모두 긍정적인 결과 보고
- 개발자 자신의 행복이 가져오는 가장 중요한 결과는
	- (빈도 측면에서) 높은 인지적 성과, 높은 동기 부여, 긍정적인 분위기
	- 높은 자기 성취감, 높은 업무 몰입도와 인내심
	- 높은 창의성, 높은 자신감
- 반대로 행복하지 않은 상태는 창의성이 감소하는 결과로 이어짐
- 프로그래머로서 배울점?
	- 휴가의 본질은 (회사의 압박 없이) 기분을 전환하고 영감을 얻는 것
	- [[CHAP6_호기심]], 어린시절의 호기심을 다시 느끼고 심지어 행복감을 느끼는 것
	- 장기 휴가, 프리랜서로 일하기, 팀 전환, 아르바이트, 다른 분야 일하기, 블로그 개설, 책 쓰기 등
		- 과감하거나, 덜 과감한 방법으로도 경험 가능
- 꼭 '안식년'일 필요는 없음
	- 안식년이라는 용어가 주는 느낌은 약간 침습적이기도, 또는 완전히 불가능한 것처럼 들릴 수 있기 때문
	- 휴가를 제한적으로 사용하거나 주말을 연장하여 일과 완전히 분리하는 것만으로도
		- 창의적 효율성이 재충전되는 것으로 입증됨
	- 조직 심리학자들은 업무로부터의 정서적, 육체적 분리가
		- 직원의 건강은 물론 창의성에도 긍정적인 영향을 미침
	- 아무리 짧다고 하더라도, 새로운 것을 발견하고 새로운 사람들을 만나기 위해 휴가를 보내는 행동은
		- 창의성과 건강 모두에 도움이 됨
- 그러나 업무에서 완전히 인지적으로 분리되면
	- 창의적 사고에 도움이 될 수 있는 업무 자원(가까운 동료, 사용 가능한 업무 자원 등)과 떨어지게 되고
	- 이것이 오히려 학습과 창의성에 부정적인 영향을 줄 수도 있음
- 저자들은 건강과 창의적 문제 해결 능력의 균형을 맞추기 위해
	- 직원들이 직장에서 휴식을 취해야 한다고 결론을 지음
	- CHAP9에서 웰빙과 창의성에 대한 이야기를 할 예정

# 8.3. 선택: 작가의 도구 상자
- 예술가 다음으로 작가들의 작가도구를 들여다보는 것 -> 유익한 정보
- 작가의 도구 상자
	- 장애물이나 두려움의 대상인 빈 페이지 극복
	- 아이디어 사이를 참신하게 연결하는 효과적인 기법
	- 약간의 수정만 거친다면 위 기법들도 창의적 프로그래머의 손에서 훌륭한 도구가 될 수 있음
- 창의적 글쓰기 기법
	- 은유, 수사학적 질문, 직유, 의인화, 자유 형식 글쓰기
	- 받아쓰기 및 필사, 행위 구조화, 줄거리 전개, 다양한 사용자 그룹에서 특정 언어 추출 -> 데이터 마이닝
	- 20억 개의 결과를 인터넷에서 찾을 수 있음
- 끝없는 글쓰기 기법 목록 -> 위대한 작가들이 공식화한 글쓰기 조언보다 더 높은 설득력 x
	- 이후 소개할 기법은 코드 작성을 창의적으로 개선하는데 도움이 되도록 작가들의 조언 몇 가지 선별

## 8.3.1. [[블라드미르 나보코프]]의 도구 상자
- 나보코프: 다양한 인터뷰, 강연에서 언제나 조언을 아끼지 않음
- Robert Golla가 편집한 Conversations With Vladimir Nabocov 책
	- 나보코프와의 인터뷰를 모아 러시아계 미국인 문학 거장의 삶과 작품을 전체적으로 소개
	- 다음은 프로그래밍 관점에서 적합하다고 생각되는 조언 선별
- **다른 예술가들을 연구하기**
	- 창의적인 작가는 창조주를 포함해 자기 라이벌의 작품을 주의 깊게 연구해야 함
	- 그는 주어진 세계를 재조합할 뿐만 아니라 재창조할 수 있는 타고난 능력을 갖추고 있어야 함
	- 이를 적절히 수행하고 노동의 중복을 피하기 위해 예술가는 주어진 세계를 알아야 함
	- 예술가처럼, 작가처럼 훔치기
- **주변 세계에서 영감을 얻을 것**
	- 글쓰기가 다른 무엇보다도 이 세상을 허구의 가능성으로 보는 예술을 의미하는 것이 아니라면 글쓰기는 쓸모 없는 일
	- 물론 소설과 소프트웨어의 비즈니스 요구사항을 구현하는 것에는 큰 차이
		- 그렇다고 IT 외부의 세상을 바라보는 것으로부터 영감을 얻을 수 없다는 의미는 아님
	- e.g. 우편물을 배달하는 우편 시스템 -> RabbitMQ, Apache Kafka
- **모든 작가는 위대한 사기꾼이자 마법사**
	- 나보코프는 소설을 쓰는 일을 자연의 단순성에 비유
		- 자연이 단순한 것처럼 보이는 것은 우리의 착각
	- 이 비유는 프로그래밍에도 적용 가능
		- 간단한 것처럼 보이는 코드를 작성하기
	- 나보코프의 말을 빌려 표현하자만,
		- **이런 코드는 엄청나게 정교한 착각을 불러일으킬 정도로 단순하면서도 그 의도 명확히 전달**
	- 스타일과 구조는 훌륭한 아이디어를 구현하는 것보다 중요하며
		- 이런 개념은 최근 고 프로그래머들 사이에 많은 지지를 얻고 있음
- **세부 사항에 신경 쓰기**
	- 많은 연필 자국에서 볼 수 있듯이, 나보코프는 인덱스카드를 끊임없이 수정함
	- 그가 "모든 카드를 여러번 다시 썼다"라고 말했듯이
		- 그의 카드에서 수정되지 않은 단어는 단 하나도 없었음
	- 프로그래밍을 할 때 단위 테스트에서 초기 요구사항을 충족하면 계쏙 수정해야 함
	- 테스트를 통해 필요한 기능이 구현되고
		- 다른 사람이 코드를 쉽게 읽을 수 있을 때까지
		- 테스트를 포함해 모든 코드 줄을 여러번 다시 작성하고 유지보수와 추후 재작업을 용이하게 해야 함
	- 레드, 그린, 리팩터링을 반복해야 함
	- **작가들의 표현 -> 당신이 사랑하는 것을 죽여야 함**
- **스타일을 개발하기**
	- 스타일은 도구도, 방법도, 단어를 선택하는 것도 아님
	- 이 모든 것 이상으로 스타일은 작가가 가지는 개성의 본질적인 구성 요소 또는 특성
	- 따라서 스타일을 말할 때 우리는 개별 예술가의 독특한 성격과
		- 그것이 예술적 결과물에서 표현되는 방식을 의미
	- "문학 경력을 쌓는 과정에서 작가의 스타일이 더욱 정확하고 인상적으로 변하는 것은 드문 일이 아님"
	- 프로그래밍은 글쓰기보다 협업 활동에 가깝지만,
		- 바로 그 점 때문에 나보코프가 이 지점에서 뭔가 시사해 주는 바가 있음
	- 우리의 (집단적) 코딩 스타일은 알고리즘이나 프레임워크의 선택 만큼이나 중요
- **반드시 처음부터 시작하지 않아도 된다**
	- 유연한 인덱스카드 시스템 덕분에 나보코프는 1장에서 시작하지 않고도 여기저기서 몇 문장을 쓸 수 있었고,
		- 2장까지 성실하게 이어갈 수도 있었음
	- 복잡한 부분을 코드로 구현할 때는 이 조언에 귀를 기울이기
	- '인덱스 카드'로 나누듯이 단위 테스트를 통해 큰 덩어리를 별도의 작은 단위로 나누면
		- 매번 처음부터 다시 시작할 필요 없이 문제를 해결할 수 있음

## 8.3.2. 제프 다이어의 도구 상자
- 영국 작가 [[제프 다이어]]는 소설과 논픽션을 모두 집필, 작품으로 많은 상을 받음
- [[블라드미르 나보코프]]보다 보다 실용적인 글쓰기 조언
- **공공장소에서 글을 쓰지 말 것**
	- 용변을 볼 때처럼 사적인 공간에서 글쓰기를 해야한다
	- 칼 뉴포트의 딥 워크와 유사
	- 글을 쓸 때는(코딩할 때는) 문을 닫아놓고, 수정할 때는 문을 열어 놓음
	- 코딩에는 집중력이 필요함
	- 커피숍에서 일하는 모습은 이상적으로 보일지는 몰라도 집중력을 높이는 데는 도움이 되지 않음
	- 다이어의 말이 맞는 다면 이런 공간은 오히려 집중을 방해함
- **자동 수정 설정을 지속해서 개선하고 확장하기**
	- 작가는 더 창의적인 생각을 위한 공간을 마련하도록 생산성 도구의 사용을 공개적으로 옹호해야 한다는 의미
	- IDE 단축키에 대한 근육 기억을 쌓아두기
	- 자동 제안 기능을 미세 조정하기
	- 코드 편집기는 가장 친한 친구
- **일기를 쓰기**
	- 파울로 코엘료와 같은 작가들은 일기의 장점을 부정함
		- "메모는 잊어버리고, 중요한 것은 남고, 중요하지 않은 것은 사라짐"
	- 하지만 현대 프로그래밍의 경우, 우리가 주의를 기울여야 하는 많은 세부사항도 사라지는 경향이 있기 때문에 이는 매우 위험한 관점
	- 다이어의 조건을 따르거나 CHAP2를 다시 읽어 볼 것
- **너무 어렵다면 포기하고 다른 일을 할 것**
	- '글쓰기는 인내의 문제'라고 다이어는 설명함
	- '계속 밀고 나가야 함'
	- 장애물을 무의식적으로 처리하는 것은 일시적으로 다른 일을 할 때 얻을 수 있는 추가적인 이점
- **습관을 만들기**
	- 매일 글을 쓰기. 인내의 문제
	- 점차 본능이 될 것
	- 몇몇 작가들의 글쓰기 스케줄은 정말 인상적
	- e.g. 스티븐 킹은 매일 만 단어를 쓴다고 함
		- 우리 시대에 가장 다작을 하는 미국 소설가 중 한명
	- 다른 작가들은 문장 몇 개를 공들여 수정하는 정도에 그침
	- 양이 중요한 것은 아니나, 매일 습관을 들이면 창의적인 성공으로 이르는 길을 더 쉽게 만듦

## 8.3.3. 앤 라모트의 도구 상자
- [[앤 라모트]]의 쓰기의 감각은
	- 작가의 고통, 은혜, 사랑, 두려움을 드러내는 데 있어 고전적인 작품
- 나보코프와 다이어의 조언에 비해 라모트의 조언은 좀 더 개인적이고 감정적
	- 하지만 그 중요성은 덜하지 않음
- 라모트의 조언 중 특히 도발적인 몇 가지
- **자신만의 독특한 스토리를 작성하기**
	- "여러분에게 일어난 일은 여러분 자신의 것. 여러분의 이야기를 들려주세요" -> 라모트의 글
	- 현직 자바 세계에 뛰어든 전직 파이썬 프로그래머 -> 파이썬 실력을 발휘하는 것을 두려워하지 말 것
	- camelCasing, snake_casing과 같은 것이 아니라
		- 개인의 과거 경험을 현재의 코딩 문제에 적용하는 창의적인 방법에 대한 것
	- 다른 누구도 아닌 여러분만이 할 수 있음
- **그냥 앉아 있기**
	- 습관 만들기에 대한 다이어의 조언을 되새기며 "매일 거의 같은 시간에 앉아 있도록 노력하기"
	- 무의식이 창의적으로 작동하도록 훈련할 수 있음
	- 영감이 떠오르는 데는 시간이 걸릴 수 있음
		- 스트레스 받지 말고 인내심을 가질 것
	- 천장과 시계를 바라보고 하품하고 종이를 다시 쳐다보기
		- 그런 다음 손가락을 키보드 위에 올려놓고 머릿속에 떠오르는 이미지(장면, 장소, 캐릭터)를 가만히 바라보며 마음을 가라앉힌 뒤
		- 그 풍경이나 캐릭터가 마음속의 다른 목소리마다 먼저 말하는 것을 들을 수 있도록 노력하기
	- **창의성 설문조사에서 일부 프로그래머는 "그냥 앉아서 글을 쓰기 시작한다"라고 답함**
	- 이들은 구문에 신경 쓰지 않고 '그냥 생각나는 것을 적기 위해' 전체적인 아이디어를 의사 코드로 입력
	- 그런 다음 다른 사람의 도움을 요청하고 점차 실행 가능한 솔루션으로 변경해 나감
- **속이지 말 것**
	- 최종 사용자는 즉시 알아차릴 것
	- 지난 몇 년 동안 우리가 아주 살짝 무뎌졌거나 작은 실수를 저질렀다 해도,
		- 독자들은 늘 현명하고 세심하다고 가정해야 함
	- 그들을 속이려고 하면 여러분의 속임수를 금방 알아차릴 것
	- 라모트는 독자들이 현명하고 세심하다고 언급
	- 프로그래밍의 경우 이는 최종 사용자에게만 국한되지 않음
	- 여러분보다 훨씬 더 많이 코드를 읽게 될 현재, 미래의 동료들도 여러분의 코드에 녹아 있는 불필요한 허풍에 속지 않을 것
	- **창의적인 프로그래머는 최종 사용자를 고려함**
		- 가능하면 직접 만나서 그들과 그들의 세계에 대해 알아보고
		- 비즈니스 요구사항을 신중하게 분석할 것
	- 반쪽짜리 고객 프로필과 비즈니스 로직에 대해 비판적이어야 함
	- 그런 다음 고객이 정말 필요로 하는 것을 코드로 작성하기
- **주변 사람들에게 도움을 요청하기**
	- 예시: 정원에 대한 정확한 설명이 필요해 이에 대해 도움을 받았던 열정적인 정원사
	- '글에 생동감을 불언허기' 위해 다른 사람의 전문 지식을 활용
		- 음악 스트리밍 웹서버를 구현해야 한다면, 음악가로부터 음정 및 일반적인 파형에 대해 배우고
		- 이를 바탕으로 데이터를 효율적으로 인코딩 하는 방법을 더 잘 이해하는게 도움이 됨
- **단계별로 작성하기**
	- 때로는 어디로 향하는지 몰라도 괜찮음
	- 최종 목적지만 고집스럽게 집중하다 보면 그 과정에서 지나가는 모든 것을 놓침
	- 소설을 쓰는 것은 밤에 자동차를 운전하는 것과 같음
		- 우리는 헤드라이트가 비추는 곳까지만 볼 수 있지만
		- 그럼에도 불구하고 목적지까지 갈 수 있음
	- **앞을 조금만 보는 것만으로도 충분함**
- **완벽을 추구하지 말 것**
	- 완벽주의는 억압자의 목소리이자 민중의 적
	- 그것은 당신을 옴짝달싹 못 하게 하고 평생 미치게 만들 것
	- 비록 형편없는 초안이라고 그것을 작성하는 데 방해가 될 것
	- 완벽주의는 각 디딤돌을 하나하나 제대로 밟으면서 충분히 조심스럽게 달리면 죽지 않을 것이라는 강박적인 믿음에 기인
		- 하지만 실제로 여러분은 죽을 것
		- 자신의 발을 쳐다보지도 않는 많은 사람이 당신보다 훨씬 더 잘할 것
		- 그러는 동안 훨씬 더 재미있게 달릴 것
	- 때로 우리의 독단적인 자아는 전혀 필요 없는 지점까지 코드를 계속 변경하고 리팩토링 하도록 스스로를 몰아갈 수 있음
	- 너무 성급하게 처리하지 않도록 **실용적인 자아**를 주의해야 하나
		- 완벽주의에 빠지지 않도록 독단적인 자아 역시 주의해야 함
	- 비판적 사고에 관한 5장의 조언, 즉 창의성은 목표가 아니라 수단임
### EXERCISE
- 새로운 동료가 코드 작성에 대한 조언을 구한다고 상상
- 어떻게 대답하겠는가
	- 그 동료에게 팀의 코딩 스타일 가이드라인을 알려주는가
	- 아니면 앉아서 개인적인 성공 스토리를 들려주는가
	- 로버트 C.마틴의 클린 코드를 읽어보라 하는가
	- 아니면 대답을 미루고 이번주에 페어 프로그래밍을 하자고 제안하는가


# 8.4. 선택: 프로그래머의 도구 상자
- 창의성 연구를 통해 환경 분석, 문제 인식 및 식별, 가정, 대안 생성 등 100가지가 넘는 다양한 창의적 문제 해결 기법이 나옴
	- 문제는 이러한 기법 중 프로그래밍 분야에 적용할 수 있는 기법이 있는가
- 소프트웨어 개발자는 자신이 생각하는 것보다 더 많은 창의적인 기술을 알고 있음
	- 에스더 더비(Esther Derby)와 다이애나 라슨(Diana Larsen)의 훌륭한 애자일 회고록 또는 관련 웹사이트를 통해
		- 이러한 기법들이 스프린트를 회고할 때 지속해서 적용되는 것을 봄
	- 그런데 왜 회의실 밖에서는 이러한 기법을 활용하지 않았는가
	- 활력 넘치는 회고를 변형한 것
		- e.g. [[트리플 니켈]](Triple Nickel) 등을 개발에 적용하기 어려운 이유는 무엇인가
- 다음 절에서는 과소평가 되거나 흔히 오해하는 여러 가지 창의적인 기법을 전문가들의 사례를 통해 소개
	- 녹슨 프로그래머의 도구를 윤기나게 만들기
	- 녹을 제거하려고 식초에 오래 담가 두는 것과 같은 일은 필요 x


## 8.4.1. 안나 밥코프스카의 도구 상자
- 2019년, 소프트웨어 공학 연구원 안나 밥코프스카(Anna E. Bobkowska)
	- 교육-응용-피드백이라는 특정 주기를 활용해 소프트웨어 공학에서 창의적인 기법의 잠재력 연구
- 실험 참가자들, 실험이 끝난 후 창의성 기법에 대한 인식이 높아짐
	- 이러한 기법을 혼합하면 실제로 유용할 것이라 주장
- 7가지 기법
	- 숨겨진 가정과 암묵적 지식을 발견하기 위해 **순진한 질문** 던지기
		- e.g. 이 웹페이지에 입력 양식이 하나만 필요하다고 상상하기
	- 만약 어떤 일을 한다면 어떤 결과가 발생할지 또는 아직 드러나지 않은 결과 찾기
		- e.g. 제출 버튼을 20번 누르면 어떻게 될까?
	- 개인적인 장애물을 이해하기 위해 '만약 ~라면 좀 더 창의적이 될 것 같아요'라는 문성 완성하기
		- e.g. 생각하기 위해 산책하는 행동을 다른 사람이 못마땅하게 보지만 않는다면 더 창의적으로 일할 수 있을 것 같아요
	- 루네트(Lunette) 기법 사용하기
		- 다양한 수준의 추상화를 통해 문제 바라보기
		- e.g. 일반화와 전문화 사이 전환, 조감도를 적용하여 코드를 확대하고 다시 축소
	- 역 브랜스토밍(Reverse brainstorming) 사용하기
		- 먼저 비판하고 그다음에는 개선할 수 있도록 동기 부여
		- e.g. 데이터베이스 구조에서 어떤 점이 마음에 들지 않나요?
	- 밥코프스카가 **중국어 사전이라고 부르는, 오래된 동물 분류법에서 파생된** 비정형 분류를 만드는 기법 사용하기
		- e.g. 프로젝트와 관련된 문제에 대해 특이한 분류 만들기
	- '초대하자' 기법 사용하기
		- 가상의 창의성 분야 전문가의 창의성 패턴 활용
		- e.g. Linus Tovalds를 초대한다고 가정할 때, 그가 이에 대해 어떤 말을 할까요?
- 상상력을 마음껏 발휘하는 것이 감춰진 제약 조건과 가정을 발견하는 훌륭한 기법으로 자주 언급됨
	- 다른 각도에서 접근하기, 시스템을 깨뜨릴 수 있는 모든 방법과 시스템이 잘못 작동할 수 있는 모든 방법을 찾아내기
	- 해결책이 있을 때, 이 것이 제대로 작동하지 않았을 때 어떤 일이 일어나는지
		- 작동하지 않더라도 문제는 없는지
		- 그리고 문제가 있다면 이를 처리할 수 있는 다음 해결책은 무엇인지
- '왜?'라고 반복해서 묻는 것도 대화나 독백에서 흔히 볼 수 있는 주제
	- 자신의 생각을 큰소리로 말하는 것을 과학자들은 **자기 주도적 말하기**라고 함
	- 시각 처리 중에 뇌가 더 나은 성능을 발휘하는데 도움이 되는 것으로 입증됨
	- 누군가 다른 사람, 예를 들어 동료나 다른 대상에게 문제를 설명하면
		- 해결책이 저절로 떠오르는 경우가 많음
	- 그 대상이 [[고무 오리]] 또는 벽에 걸린 편리한 예술 작품(e.g. 미술 기반 학습에서)이나 고양이라 해도 무관
	- 설명하거나 심지어 가르치다 보면 속도를 늦추고 다양한 방향에서 문제에 접근하게 됨, 대게는 더 깊이 이해
- 대부분의 기법은 구현과 관련된 장애물을 식별하고 해결하는데 유용한 도구이나
	- 역 브레인 스토밍이나 비정형 분류와 같은 기법은 아이디어 구상 단계와 같이 더 이른 시점에 사용 가능
	- 밥코프스카는 이들 기법을 4가지 주제로 분류
		- **대인관계 기술**(팀 정신 형성, 개인적인 장애물 제거)
		- **창의력 기술**(연상적 사고 및 아이디어)
		- **동기 부여 기술**(부정적인 측면을 발견하는 데 도움)
		- **장애물 극복**

### 브레인스토밍의 나쁜 점
- 창의적 컨설턴트들이 추천하는 브레인스토밍 세션만큼 고정관념에 치우친게 있겠는가
	- 브레인스토밍은 연구들에게 가장 자주 언급되는 방법
- 하지만 몇 가지 정리 필요
	- 많은 결과에 따르면 브레인스토밍은 효과가 없다는 것이 입증됨
	- 사람들은 회실에서 화이트보드 앞에 앉아 있을 때보다 혼자 있을 때 두 배나 더 많은 아이디어를 만들어냄
	- 우리 모두가 얻고자 하는 하나의 '큰' 아이디어를 위해 점을 연결하는 작업도 혼자 있을 때 더 잘함
	- 많은 브레인스토밍 세션에는 어떤 의도가 숨겨져 있을 때가 있고
		- 이로 인해 정말 좋은 아이디어를 억누르는 경우가 많음
	- 무심결에 툭툭 내뱉는 말이 다른 사람들의 사고 과정에 영향
		- 우리 주변에 '그건 절대 안될꺼야'라고 말하면서 고개를 저으며 우리가 말하는 것을 방해하는 동료는 한 두명씩 있기 마련
	- [[CHAP3_Communication]]에서 언급했듯 집단 지성은 아이디어가 충분히 이질적일 때만 작동 가능
- 그러면 브레인 스토밍을 포기해야 하는가?
	- 서로를 존중하는 집단 모임은 여전히 아이디어를 모으는 훌륭한 방법
	- 회사 회의실이 아닌 커피숍 등



## 8.4.2. 실용주의 프로그래머의 도구 상자
- 프로그래머의 바이블 격인 '실용주의 프로그래머'는 매년 새로운 프로그래밍 언어를 배울 것을 권장
	- 각각 새로운 언어는 그 언어만의 고유한 지침, 스타일, 열렬한 추종자, 문제 해결에 대한 고유 접근 방식이 있음
	- 4장에서 설명한 대로 언어 간 충돌을 염두에 두어야 하지만
		- 더 많은 언어를 접할수록 한 언어의 관행을 창의적으로 결합하고 다른 언어로 변환할 수 있는 가능성이 커짐
- 브루트 테이트(Bruce Tate)는 앤디 헌트와 데이비드 토머스의 추천이 마음에 들지 x
	- 1년에 한 언어만 배우기?
	- 7주 동안 7개 언어를 배우는건 어떤지
	- 실제로 '브루스 테이트의 세븐 랭귀지'에서 테이트는 두 달 동안 7개의 언어를 배우는 여정을 바탕으로 새로운 프로그래밍 언어를 빠르게 배우기 위한 실용적인 팁 제공
	- 이 책은 큰 성공, 4년 후에 'Seven More Languages in Seven Weeks' 후속작으로 이어짐
	- 테이트의 책은 언어의 기술적인 측면을 다루고 있으며, 독자들에게 다양한 커뮤니티의 다른 프로그래머들이 복잡한 문제를 어떻게 해결하는지를 맛보게 해줌
- 여러 언어의 관점에서 생각하는 것은 코드의 어려운 문제에 접근하는 효과적인 방법
	- 어떤 문제에 봉착했는데 현재 기술로는 어떻게 해결해야 할지 모른다면?
	- 하지만 자바스크립트로 작성할 수 있다면
		- 혹은 엘릭서나 코틀린으로 작성한다면
	- C의 포인터에 의존하거나 루비의 리플렉션 확장을 활용하거나 함수형 언어를 표현한다면?
	- 그렇다면 일이 더 쉬워지는가 혹은 어려워지는가
	- 다른 언어의 Null type은 어떤가
	- 비즈니스 로직을 프롤로그로 표현해야할지
	- Squirrel과 같은 스크립팅 언어의 장점을 활용할지
	- 로직을 표현하기 위해 사용자 지정 도메인 전용 언어는 어떨지
- 현재 언어로 할 수 없다면 다른 언어를 시도해보기
	- 그 언어로 안 되면 또 다른 언어로 시도하기
	- 7번째 시도라면 아이디어를 Backporting 해보는 것으로 ㅊ우분
	- 가상 머신이 해당 언어 해석 가능
	- 루비와 파이썬은 JVM과 CLR로 실행되고
		- Clojure는 JVM에서 실행되는 Lisp의 한 종류,
		- JS는 요즘 어디서나 수행 가능
- 많은 프로그래머가 자신의 일상 업무, 사고 방식에 갇혀있음
	- 이런 틀에 박힌 시야 때문에 또 다른 가능성을 발견하는 과정이 방해를 받음
	- 그리고 또 다른 가능성은 종종 더 나은 가능성이기도 함
	- 흥미롭게도 포커스 그룹 참가자 중 언어를 바꾸거나 다른 언어의 패턴에 대해 생각하는 일을 창의적인 활동으로 언급한 사람은 없었음
	- 대신, 인터뷰에 참여한 사람들은 구문과 씨름하는 것을 창의적이지 않은 활동으로 꼽음
- 창의적이지 않은 활동?
	- 구문 오류, 단순히 문법에 맞춰 작업하기
	- 테스트의 기계적 오류
	- 새로운 기술을 사용하는 경우 해당 기술에 대한 문서를 읽거나
		- 그 기술은 어떻게 작동하는지, 그 프로토콜은 어떻게 작동하는지 등을 살펴보는 것
		- **이 부분은 창의력을 발휘하는데 필요한 기술 지식의 기준 암시**
- 새로운 언어를 매년 배우는 것 뿐만 아니라, 소프트웨어 개발 세계에 무술 katas 개념을 소개함
	- [[코드 카타 연습]]

## 8.4.3. 에밀리 모어하우스의 도구 상자
- 2015년, Emily Morehouse는 첫 번째 파이콘 컨퍼런스에 참여
	- CPython, 550k 줄 이상의 C 코드, 629k 줄 이상의 python 코드 포함
	- 프로젝트에 기여하려면 어디서부터 시작해야할까?
	- 소스코드를 공부하기
		- 다른 핵심 개발자들이 어떻게 작업하는지
		- 어떤 패턴이 반복적으로 적용되는지
		- 의사 결정은 어떻게 이루어지는지
		- 개선이 필요한 부분들
- 다른 사람들의 소스코드를 연구 -> 오스틴 클레온의 '훔쳐라, 아티스트처럼' 선언문 처럼 자신의 작품에 사용할 귀중한 교훈을 얻을 수 있음
	- 소설가에게 조언을 구하면 가장 먼저 하는 말이 '더 많이 읽으라'
	- 좋은 작가가 되려면 먼저 읽어야 함
- 일상적인 프로젝트 코드베이스 외에 다른 사람의 코드를 의도적으로 읽고 배우는 일이 얼마나 적은가
	- 프로그래밍에 대한 책은 다루나, 무료 오픈 소스 소프트웨어(FOSS) 코드를 다룬 적은 x
	- 좋은 프로그래머가 되려면 먼저 코드를 읽어야 함
- 에릭 레이먼드(Eric S. Raymond)
	- 유명한 해커이자 오픈 소스 소프트웨어 옹호자
	- '해커가 되는법'에서 제안한 내용 -> 코드가 아닌 글도 써보기
	- 코드가 아닌 텍스트, 글쓰기에 대한 예제는 CHAP1에 존재
- Github, Gitlab과 같은 소셜 코딩 플랫폼은 FOSS 코드를 읽고 이해하는데 큰 도움이 됨
	- 많은 대형 프로젝트에서 적극적으로 기여자를 찾고 있으며, 끝없이 쏟아지는 이슈와 코드 줄이 버거울 수 있으나
	- FOSS에 기여하는 것은 자신만의 창의적인 프로그래밍 툴킷을 연마할 수 있는 좋은 방법
- 파이썬 코어 개발자가 되고 싶다는 에밀리 모어하우스 처럼 진지하게 기여하고 싶다면
	- 자신의 작업을 점검해줄 멘토를 갖는 것은 불필요하지 않음
	- 에밀리는 누군가의 지도가 없었다면 포기했을 것이라고 인정
	- 어떤 맥락이 절실히 필요함에도 무슨 수를 써도 보이지 않는 상황에서 멘토는 보여줄 수 있음
	- 코드의 어떤 부분은 누구도 감히 건드릴 수 없는 유물이 되기도 함
		- 왜 이 파일에 이상한 C 함수가 그룹화 되어 있는지 등
	- 적절한 문서화가 없다면 이러한 집단적 지식은 시간이 지나면서 빠르게 사라짐
- 모어하우스는 코드를 읽고 멘토 갖는 것 외에 개발 커뮤니티의 신뢰를 얻는 것이 중요하다고 말함
	- 이는 시간이 오래 걸리는 과정
	- 공감대 형성에 대한 것은 그 어떤 `CONTRIBUING.md` 가이드에서 언급된 적 x
	- 성공적으로 녹아든 후에 새로운 기여자를 도와주고 인터넷상에서 귀찮게 하는 익명의 사람들을 상대하는 건 독자의 몫
- 파이썬 코어 개발자는 다른 개발자와 마찬가지로
	- 단순히 코드 작성하는 것외에 다른 많은 일을 함

# 8.5. 요약
- 복잡한 문제에 접근할 때는 때때로 도구를 전환할 것
	- 문제를 세부적으로 들여다보지 말고 조금 멀리 떨어져서 다른 관점으로 문제를 바라볼 것
- 창작 도구 상자에 있는 도구를 잘 관리할 것
	- 때때로 비판적으로 평가하고 다시 연마할 것
	- 무딘 도구는 버려야 함
- 기술 이외의 명작을 감상하면 아이디어가 떠오르고 그 개념을 코드에 반영하는데 영감을 얻을 수 있음
	- 창의성에 관해서는 자신의 생각에만 귀를 기울이지 말고 명작이 말하게 할 것
- 아이디어를 비하하거나, 대충 훑어보거나, 무시하지 말 것
	- 대신 아이디어를 존중하고, 연구하고, 리믹스 할 것
	- 사기꾼이 아닌 예술가처럼 훔칠 것
- 자신의 가려운 곳을 긁을 것
	- 샛길을 탐구하는 것은 창의적인 문제 해결 능력을 향상하고
	- 잠재적인 신제품을 만드는데 효과적
- 코드를 작성할 때는 키보드에 백스페이스가 있다는 것을 기억할 것
	- **때로는 무언가 더하는 것보다 빼는 것이 코드를 더 명확하게 함**
- 경력을 쌓는 데 다운타임을 활용하고 때때로 휴식을 취해 창의적인 사고의 베터리를 재충전 할 것
	- 일상 업무에서 벗어나 새로운 기술을 탐구하고 새로운 사람들을 만나면
	- 여러분의 업무는 영감을 얻고 발전할 수 있음
- 현재의 기분과 감정이 인지능력, 즉 창의적인 문제 해결 능력에 긍정적인 혹은 부정적인 영향을 미칠 수 있음
	- 매일 창의적인 하루를 보낼 필요는 없음
- 코드에 있어 스타일과 구조는 콘텐츠와 기능적 정확성만큼이나 중요함
	- 자신의 코드에서 창의성을 너무 과장하다 보면 정작 코드의 유지보수성은 나빠질 수 있음
- 처음이 어렵다면 중간부터 시작하거나 심지어 마지막부터 시작할 것
	- 구문이 어렵다면 구문 오류는 무시하고 그냥 작성할 것
	- 아이디어의 기본을 먼저 파악하면 대상 프로그래밍 언어로 구현하기가 더 쉬워짐
- 프로그래머에게 창작 도구 상자에서 가장 중요한 도구 중 하나는 코드 편집기
	- 코드 편집기에 대해 잘 알아야 함
	- 단축키와 다양한 설정에 익숙해지는데 시간을 투자할 것
	- 기본 사항을 숙지하면 큰 도움이 됨
- 장애물에 부딪혔을 때 일시적으로 포기하고 다른 일을 하는 것도 괜찮음
	- 한 시간이나 하루가 지나면 해결책을 찾을 수 있음
- 특정 프로그래밍 언어에 대한 개인적인 이력을 잘 활용할 것
	- 루비의 메시지 전달 구문을 잘 알고 있다면 엘릭서의 메서드를 사용하는데 큰 어려움이 없을 것
	- 그 연장선상에서 다음 달까지 두 가지 새로운 언어를 배우는 데 도전할 것
- 좋아하는 애자일 브레인스토밍 도구를 전통적인 회의실 밖에서 사용할 것
- 부모님, 자녀, 프로그래머가 아닌 친구, 고객과 함께 코딩 문제에 대해 토론할 것
	- 비전문가인 이들의 시각을 통해 전문가인 여러분이 간과한 간단한 해결책 발견 가능
- [[코드 카타 연습]]을 수행하려고 한다면 현재 프로젝트의 소스 저장소에서 코드를 분리할 수 있는지 확인할 것
	- 아마도 이러한 코드는 기존에 전통적으로 사용해 온 예제보다 여러분과 동료들을 교육하기에 더 좋을 것
- 아직 멘토가 없다면 자신만의 창의적인 도구를 만드는 데 도움을 줄 수 있는 멘토를 찾을 것
